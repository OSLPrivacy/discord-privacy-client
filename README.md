# OSL Privacy

[![Rust Test](https://github.com/OSLPrivacy/discord-privacy-client/actions/workflows/rust-test.yml/badge.svg)](https://github.com/OSLPrivacy/discord-privacy-client/actions/workflows/rust-test.yml)

OSL is a desktop Discord client that adds end to end encryption on top of
normal Discord. You log in to Discord the way you always do. When you turn on
encryption for a conversation, the messages you send are encrypted on your
machine before they ever reach Discord's servers, and they are decrypted again
only on the machines of the people you chose to share with. Discord stores and
forwards an unreadable cover instead of your text.

This is Beta 1.0. It works for direct messages, group chats, and server
channels, including images and edits. It has been tested between a small number
of machines. Treat it as beta software: keep a backup plan and do not rely on it
for anything where a failure would be serious.

This README describes the original Discord-specific client. The newer
multi-service standalone shell under `apps/osl-hub*` is a separate pre-release
preview; its service-neutral encrypted transport and native composer adapters
remain fail-closed until their recipient and composer bindings are proven.

## What it protects and what it does not

OSL hides the content of your messages from Discord, from your network, and from
anyone who later gets hold of Discord's copy. It does not hide that you are using
Discord, who you talk to, or when. Someone watching the account level still sees
that two accounts exchanged messages at a certain time. If your threat model
includes a targeted investigation of you specifically, use Signal, Briar, or
Cwtch instead. OSL raises the cost of bulk collection and casual snooping on
Discord. It is not a witness protection program.

The full write up is in [`docs/THREAT_MODEL.md`](docs/THREAT_MODEL.md).

## How the encryption works

Every install generates a long term identity on first launch. The identity holds
an X25519 key, an Ed25519 signing key, and an ML-KEM-768 key for post quantum
protection. The public halves are published to the keyserver under your Discord
ID so other people can find them. The private halves never leave your machine and
are sealed at rest (TPM where available, OS keychain otherwise).

When you message someone, the two identities run a hybrid handshake (PQXDH) that
mixes classical X25519 with ML-KEM-768, so a future quantum computer that breaks
X25519 still cannot read the traffic. Direct messages then ride a Double Ratchet,
which gives forward secrecy: stealing today's keys does not unlock yesterday's
messages. Group chats and server channels use sender keys, where each sender has
a per group key that is distributed to the other members in an encrypted bundle,
so a message is encrypted once instead of once per person.

The encrypted bytes do not get posted to Discord as an obvious blob. They are
uploaded to a holding store and the message that actually appears in the channel
is a short piece of ordinary looking chat text generated by a language model.
Only someone with the conversation key can tell that the chat text is really a
pointer to an encrypted message and fetch the real thing.

Control traffic (the key bundles, burn notices, recovery requests) does not go
through Discord at all in Beta 1.0. It goes through the keyserver inbox, so the
channel stays clean and Discord never sees it.

## Using it

### The lock button

Every channel header gets a lock icon. It has three states:

- Open (grey): nobody in this conversation is set up, messages go out as plain
  text.
- Partial (yellow): some people are set up, messages are encrypted for them and
  anyone not set up sees the cover text.
- Full (green): everyone is set up, the whole conversation is encrypted.

Click the lock to change the encryption setting for that conversation. In a
server you can encrypt a single channel or turn on a server wide setting that
covers every channel you are in.

### Burn

There are two kinds of burn and they are different.

Scope burn destroys your keys for one conversation. The messages you sent in that
conversation become unreadable to everyone, including you, and a burn notice goes
to the other members so their copies go dark too. Use this when you no longer
trust the people in one channel with your past messages. Everything else you have
is untouched.

Account burn destroys everything. Every key, every conversation, every saved
message on your machine. It generates a fresh identity so you can start over. Use
this as a panic button. It cannot be undone.

### Settings

Open settings from the gear in the OSL toolbar. You can manage your identity, set
a password and recovery phrase, pick which servers and channels encrypt by
default, choose an update channel, and review keyboard shortcuts. Burn controls
live here too.

## Requirements

Windows 10 or newer. macOS and Linux are not supported and onboarding will exit
on them. You need a normal Discord account; OSL signs in through the real Discord
login.

## Discord Terms of Service

Using OSL may break Discord's Terms of Service and could get your account
suspended. You take that risk on yourself.

## Repository layout

```
src-tauri/            Tauri app shell (Rust) and the injected boot.js
crates/
  crypto/             PQXDH hybrid handshake, Double Ratchet, sender keys, AEAD
  stego/              Cover text generation (bigram language model + templates)
  keystore/           Identity, key sealing, keyserver client, control inbox
  store/              Encrypted local message store (SQLite)
  ipc/                Tauri command surface and message pipeline
keyserver-cf/         Keyserver on Cloudflare Workers (D1): identity, prekeys,
                      licenses, control inbox
cipher-store-cf/      Encrypted blob holding store on Cloudflare Workers
docs/
  THREAT_MODEL.md
  design/             Per feature design docs
```

## Building

The app is a Tauri 2 project. From `src-tauri/` run `cargo tauri dev` for a
development build. The two Cloudflare Workers in `keyserver-cf/` and
`cipher-store-cf/` deploy with `npm run deploy` and apply their database
migrations with `wrangler d1 migrations apply`.
